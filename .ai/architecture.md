# Architecture

A single Lovelace custom card. One root element orchestrates a circular dial
element plus a list of "feature" elements, each of which renders a small control
bound to a Home Assistant entity. Everything is a Lit web component in its own
shadow DOM; theming flows through inherited CSS custom properties.

## Stack & build

- **Lit 3** (`LitElement`, `html`/`svg`/`css` tagged templates, decorators).
- **TypeScript** (strict), compiled by `@rollup/plugin-typescript`.
- **Rollup** (`rollup.config.mjs`) → one IIFE/ESM bundle at
  `dist/material-thermostat-card.js`, minified with terser. Deps (`lit`,
  `custom-card-helpers`, `home-assistant-js-websocket`) are bundled in.
- Output is **committed to git** and attached to GitHub releases by
  `.github/workflows/release.yml`. CI (`validate.yml`) fails if `dist/` is stale.

## Entry point & registration

`src/material-thermostat-card.ts` is the entry. On module load it:

1. `registerMtIcons()` — installs the `mt:` icon set on `window.customIcons` (see
   [`icons.md`](icons.md)).
2. `console.info`s a version banner.
3. Pushes a descriptor onto `window.customCards` so the card shows in Lovelace's
   "add card" picker (with `preview: true`).
4. Defines the `<material-thermostat-card>` and (lazily) the editor element.

`getConfigElement()` dynamically imports `./editor` so the editor code isn't in
the hot path for rendering the card.

```

      └─ one of:
         mt-climate-selector ┐

         mt-switch-group     ┘

         mt-button-list       (icon chips directly)
         mt-entity-tile       (rounded tile)
```

## Data flow

The card is a **controlled component over HA state** with **optimistic** local
overrides for temperature:

1. Lovelace sets `.hass` (every state tick) and `.setConfig(config)` once.
2. `shouldUpdate()` gates re-renders: it returns true only when `_config`, a
   tracked optimistic `@state`, `_widthPx`, **or a tracked entity's state object**
   changed (`_trackedEntityIds()` collects the climate entity + every feature
   entity). ⚠️ A custom `shouldUpdate` must whitelist **every** `@state` that
   affects render or those updates silently no-op — this bit us hard (see
   [`gotchas.md`](gotchas.md)).
3. `render()` computes the layout (`_layout()` → see [`layout.md`](layout.md)),
   builds the dial bindings from the climate entity's attributes, and maps each
   `FeatureConfig` to an `<mt-feature-row>` with computed `row/colStart/span`.
4. The dial emits `value-changing` (live drag) and `value-changed` (commit). The
   card stores optimistic `_selectedTemp/_selectedLow/_selectedHigh`, then
   debounces (`SERVICE_DEBOUNCE_MS = 600`) a `climate.set_temperature` call.
   `updated()` clears each optimistic value once HA echoes it back.
5. Features call `hass.callService(...)` directly on selection (no optimistic
   layer — they reflect HA state on the next tick).

## Width measurement

`_widthPx` is the card's inner content width, measured by a `ResizeObserver` on
**the host element** (not the inner `ha-card`, which doesn't exist until both
`hass` and `config` are set). Inner width = host width − `CARD_PADDING_X` (32).
The observer is attached in `connectedCallback` (idempotent, survives
disconnect/reconnect). See [`gotchas.md`](gotchas.md) for why the host.

## Theming

`theme.ts` exports a `tokens` `css` block applied to every component's host. It
aliases Material 3 system tokens with HA fallbacks, e.g.
`--mt-on-surface: var(--md-sys-color-on-surface, var(--primary-text-color, #1c1b1f))`.
CSS custom properties inherit across shadow boundaries, so defining them on each
host cascades them into nested feature components. `climateModeColor(mode)` maps
HVAC modes to `--state-climate-*-color` vars (with hex fallbacks). The card also
honors a per-card `theme:` config via `applyThemesOnElement`.

## Release / CI

- **`validate.yml`** (push to main, PRs, weekly cron): HACS validation action
  (`category: plugin`) + lint + typecheck + build + a guard that `dist/` is
  up to date and committed.
- **`release.yml`** (on GitHub release published): builds and attaches
  `dist/material-thermostat-card.js` to the release.
- **Release procedure**: bump `CARD_VERSION` (`src/const.ts`) and `version`
  (`package.json`), `npm run build`, commit `src` + `dist` + version, push. Tags
  are created via the GitHub release UI (not `git tag` in this flow).
- `hacs.json` declares `render_readme: true` and `homeassistant: 2024.8.0` (min
  HA version). Installed via HACS as a **Lovelace/plugin** resource.

## Conventions

- ESLint flat config (`eslint.config.mjs`) + Prettier (`.prettierrc`: single
  quotes, trailing commas es5). Unused vars must be `_`-prefixed.
- JSDoc with `@param` on public methods/classes is the house style — keep it.
- Custom elements use the `mt-` prefix (except the two public ones:
  `material-thermostat-card` / `…-editor`). Each file `declare global`s its tag.
